# Code Principles

이 프로젝트는 다음의 핵심 개발 원칙을 준수하여 작성됩니다.

## 1. 명확한 변수명 (Naming)

**목적물과 행위가 확실하게 드러나는 변수명을 사용합니다.**

- 변수명만으로 해당 변수의 역할과 생명주기를 파악할 수 있어야 합니다.
- 모호한 축약어나 일반적인 명사(예: `data`, `item`) 사용을 지양합니다.
- **Good:** `userProfileImage`, `fetchTrackList`, `isRegionSelected`
- **Bad:** `img`, `list`, `flag`

## 2. 관심사 분리 (Separation of Concerns)

**각 모듈은 단 하나의 역할에 집중합니다.**

- 비즈니스 로직, 데이터 처리, UI 렌더링을 명확히 분리합니다.
- 하나의 컴포넌트나 함수가 너무 많은 책임을 지지 않도록 지속적으로 리팩토링합니다.
- 스타일, 로직, 뷰를 분리하여 유지보수성을 높입니다.

## 3. 의존성 관리 (Dependency Management)

**직접 의존을 피하고 추상화와 의존성 주입을 지향합니다.**

- 상위 모듈이 하위 모듈의 구체적인 구현에 의존하지 않도록 합니다.
- 결합도를 낮추기 위해 인터페이스나 훅(Hook)을 통한 주입 패턴을 활용합니다.
- 이를 통해 유연하고 테스트 가능한 코드를 유지합니다.

## 4. 자기 문서화 코드 (Self-Documenting Code)

**변수명과 함수명이 주석을 대신해야 합니다.**

- "이 코드는 ~을 합니다"라는 주석 대신, 그 내용을 담은 함수명으로 리팩토링합니다.
- 복잡한 조건식은 의미를 알 수 있는 변수로 추출하여 가독성을 높입니다.
- 주석은 '무엇(What)'이 아닌 '왜(Why)'를 설명할 때만 제한적으로 사용합니다.

## 5. 지속적인 리팩토링 (Continuous Refactoring)

**코드를 더 효율적이고 유지보수하기 쉬운 상태로 지속적으로 개선합니다.**

- 중복된 코드를 함수나 컴포넌트로 추출합니다.
- 불필요한 복잡성을 제거하고, 코드의 가독성을 높입니다.
- 작은 단위로 자주 리팩토링하여 기술 부채를 최소화합니다.

## 6. 테스트 가능한 설계 (Testable Design)

**테스트하기 쉬운 구조로 코드를 작성합니다.**

- 의존성 주입을 활용하여 Mock/Stub으로 대체 가능하게 만듭니다.
- 단위 테스트를 작성하여 코드의 기능을 검증합니다.
- 테스트는 리팩토링의 안전망 역할을 합니다.

## 7. 에러 처리 일관성 (Consistent Error Handling)

**명확하고 일관된 에러 처리 전략을 유지합니다.**

- 커스텀 에러 클래스를 사용하여 에러를 분류합니다.
- 비즈니스 로직에서는 에러를 던지고, UI 레이어에서 처리합니다.
- 사용자 친화적인 에러 메시지를 제공합니다.

## 8. 타입 안정성 (Type Safety)

**TypeScript의 타입 시스템을 최대한 활용합니다.**

- `any` 타입 사용을 최소화합니다.
- 명시적인 타입 정의로 런타임 에러를 방지합니다.
- 타입 추론을 활용하되, 복잡한 경우 명시적으로 타입을 선언합니다.

## 9. 점진적 개선 (Incremental Improvement)

**완벽을 추구하기보다 지속적인 개선을 지향합니다.**

- 작동하는 코드를 먼저 만들고, 점진적으로 개선합니다.
- 한 번에 모든 것을 바꾸려 하지 않습니다.
- 레거시 코드와 새 코드가 공존할 수 있도록 설계합니다.

---

## 우선순위

프로젝트 개발 시 다음 순서를 권장합니다:

1. **작동하는 코드** - 먼저 기능을 구현합니다
2. **리팩토링** - 가독성과 유지보수성을 개선합니다
3. **테스트** - 안정성을 확보합니다
4. **문서화** - 필요한 경우 추가 설명을 작성합니다
5. **성능 최적화** - 병목이 확인된 경우에만 수행합니다

> "Make it work, make it right, make it fast." - Kent Beck
